Quick Start — How to Begin

Four entry points depending on where you are in the project

🚀 Starting a New Project?

Run this command in Claude Code. You'll be asked 8 questions, then receive a personalised BA Delivery Roadmap.

/ba-start
  • 1Project type — software, transformation, AI, product, government?
  • 2Delivery stage — discovery, requirements, active sprints, UAT, change?
  • 3Project / client name
  • 4Key stakeholders (2–3 names and roles)
  • 5Business objective — what changes in the business when delivered?
  • 6System scope — what platforms or modules are in scope?
  • 7Target date or key milestone
  • 8Team setup — solo BA, BA + PO, BA team, embedded in squad?
/ba-start

New Project Kickoff

Full context interview → personalised BA Delivery Roadmap → Phase 1 begins automatically.

Phase 4 — Backlog Creation

Decompose requirements into implementable epics, stories, and technical work items

🎯Gate: Full backlog written, all stories have ≥3 acceptance criteria, business rules linked → proceed to Phase 5.
📐
Decomposition hierarchy: Initiative → Capability → Epic (1–3 sprints) → Feature (logical grouping) → Story (one sprint, INVEST) → Acceptance Criteria
/ba-epics

Epic Hierarchy

Structured decomposition from initiative to story. Hierarchy sheet + Backlog sheet + Release Roadmap. Dependencies and priorities at every level.

XLSX · 3 sheets
/ba-user-stories

User Stories + ACs

INVEST stories with ≥3 Given-When-Then acceptance criteria (happy + error + edge), business rules, dependencies, assumptions, and DoR checklist.

DOCXXLSX
/ba-tech-stories

Technical Stories

System-actor stories for APIs, scheduled jobs, audit logging, migrations, integrations, and monitoring. Full logging/security/monitoring requirements.

DOCX
/ba-brd

Lightweight BRD

Modern 5–10 page BRD: problem statement, objectives, scope in/out, personas, process flows, requirements summary, constraints, risks, open questions.

DOCX

Story Format — Quick Reference

As a [specific user role],
I want to [action/capability],
So that [measurable business outcome].

Acceptance Criteria:
Given [happy path context], When [action], Then [expected outcome].
Given [error / invalid input], When [action], Then [error handling].
Given [edge case], When [action], Then [edge case behavior].

Business Rules: BR-001, BR-002
Traceability: REQ-042 → EPIC-07 → US-187

Quality Standards

Every artifact is scored before finalising. Any dimension scoring below 4 must be improved.

Quality Scoring (1–5)

DimensionWhat to Check
ClarityUnambiguous? Readable by all audiences?
CompletenessAll sections? All edge cases?
TraceabilityLinks upstream and downstream?
TestabilityCan QA write a test from this?
ConsistencySame terminology throughout?
Business RelevanceTied to measurable outcome?
Implementation ReadinessCan a developer build this today?

Sprint Readiness — Definition of Ready

☐ Scope is unambiguous
☐ ≥3 Acceptance Criteria (happy + error + edge)
☐ Business rules referenced
☐ Dependencies identified and planned
☐ Data needs defined
☐ UX / process reference available
☐ Testability confirmed
☐ Estimation possible
☐ No blocking open questions
☐ Product / business approval received

Traceability Chain

Business Objective
  → Stakeholder Need
    → Requirement (FR/NFR/INT)
      → Epic
        → User Story
          → Business Rule
            → Acceptance Criteria
              → Test Case
                → UAT Feedback
                  → Release / Change
⚠ Missing link = TRACEABILITY GAP (flagged automatically)

Naming Convention

  • Dates: YYYY-MM-DD
  • Versions: -v1.0 suffix on final
